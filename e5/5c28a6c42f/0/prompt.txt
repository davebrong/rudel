# Plan: Navigate by Display Name in Projects List

## Context

The Projects list page groups sessions by **display name** (last segment of `git_remote` or `project_path`, e.g. `"rudel"`) via `PROJECT_DISPLAY_EXPR` in ClickHouse. Each row shows this short name in the table.

However, `handleRowClick` navigates using the raw `row.git_remote || row.project_path` — the full URL or absolute filesystem path — not the display name. This causes two problems:

1. The URL contains a long opaque path that doesn't match what the user sees.
2. The detail API's `buildProjectDisplaySubquery` resolves the display name from the full path, which is fragile and may explain the ongoing 500 errors (subquery over-complexity for certain data shapes).

The fix is simple: **navigate with `row.repository`** (the display name already returned by `getProjectInvestment`). The API's `buildProjectDisplaySubquery` fallback already handles a plain display name correctly — when the input has no `/`, it returns the input as-is, and the main query becomes `WHERE PROJECT_DISPLAY_EXPR = 'rudel'`, which aggregates **all** sessions for that project regardless of how they were uploaded.

## Changes

### `apps/web/src/pages/dashboard/ProjectsListPage.tsx`

Change `handleRowClick` (line 149–153) from:
```typescript
const key = row.git_remote || row.project_path;
```
to:
```typescript
const key = row.repository || row.git_remote || row.project_path;
```

`row.repository` is `PROJECT_DISPLAY_EXPR` (e.g. `"rudel"`), already in `ProjectInvestment` returned by `getProjectInvestment`. This is the only change needed.

**Why this works end-to-end:**
- URL becomes `/dashboard/projects/rudel` instead of `/dashboard/projects/%2FUsers%2Fmarc%2F...`
- `ProjectDetailPage` decodes `"rudel"` and passes it as `projectPath` to all detail endpoints
- `buildProjectDisplaySubquery("org", "rudel")`: no session has `project_path='rudel'`, so `count()=0`, fallback runs: `position('rudel', '/')=0` → returns `'rudel'` directly
- Main query: `WHERE PROJECT_DISPLAY_EXPR = 'rudel'` — correctly aggregates ALL sessions whose display name is `"rudel"`, regardless of git_remote or project_path
